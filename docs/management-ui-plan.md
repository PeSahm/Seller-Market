# Seller-Market Multi-Tenant Management UI — Implementation Plan

## Context

You operate **Seller-Market**, a Python + Locust trading bot that buys Iranian stock-market shares ("سر خطی" high-demand opening orders) across multiple brokers (gs, bbi, shahr, ib, karamad, tejarat, ebb, hbc, rabin, ayandeh). Each broker connection is a customer account, declared in `config.ini` as `[Name_broker_symbol]` sections (`username`, `password`, `broker`, `isin`, `side`).

Today the bot is deployed via Docker (`docker-compose.yml` on each Linux server, mounted at `/root/seller-market/`). It runs a Telegram bot + an in-process scheduler (`scheduler.py`) that fires `cache_warmup.py` then `locust -f locustfile_new.py --headless` at scheduled times based on `scheduler_config.json` and `locust_config.json`. Results land in `order_results/*.json` and `trading_bot.log`.

**The problem:** you have many servers; managing them means SSH-in, hand-edit `config.ini`, comment customers in/out, restart. You also want to onboard **agents** who manage their own customers without touching anyone else's.

**The goal:** a web admin panel that lives on a separate VPS, knows about all your servers and agents, manages everything via SSH + Docker, and gives agents a scoped UI. The existing app code stays **untouched** — the management UI sits above it.

## Locked-In Decisions (already agreed)

| Topic | Decision |
|---|---|
| Isolation | Per-agent Docker stack per server (`sm-agent-<id>-bot` only, stack dir `/root/seller-market/agents/<agent_id>/`). No per-stack OCR. |
| OCR service | **Single shared remote OCR** declared globally in mgmt UI settings (e.g. `http://5.10.248.55:18080`). Every agent stack on every server points `OCR_SERVICE_URL` to this URL. **Not deployed by mgmt UI.** |
| Tech stack | FastAPI + PostgreSQL + HTMX/Jinja |
| Hosting | Separate management VPS, talks to trading servers via SSH (Paramiko) + `docker compose` |
| Scheduler | Each agent's stack has its own `scheduler_config.json`; UI writes it; existing scheduler hot-reloads |
| Customer ownership | 1:1 — each customer belongs to exactly one agent |
| Server selection | **Agents never pick servers.** They declare customers; admin (or auto-policy) distributes them across servers. Agents only see "which server is my customer on" as a read-only badge. |
| Section names in `config.ini` | **Auto-generated by mgmt UI**, globally unique, immutable. Format: `[a{agent_id}_c{customer_id}_{broker}_{isin}]` (e.g. `[a3_c42_bbi_IRO3AYHZ0001]`). ISIN is included so the same account can hold multiple sections for different symbols. Agents only enter `display_name`. Collision-free across agents, servers, and ISINs. |
| Multi-symbol per account | One `customers` row = one (account, broker, ISIN, side) tuple = one `config.ini` section. Same person can have N rows sharing username/password/broker but differing in ISIN. UI offers a "Duplicate with new ISIN" action so agents don't re-enter credentials. |
| Telegram | Agent stacks skip the bot. **The existing manual deployment at `/root/seller-market/` (root level) is left completely untouched** — same Telegram bot, same config.ini, same workflow. Mgmt UI only operates inside the `agents/<id>/` subdirectories. |
| Admin scope | Admin role can do **everything an agent can do, for every agent** — manage any agent's customers, schedules, locust params, trigger runs, view history. Plus admin-only: servers, agents, OCR URL, global audit, all stacks. |
| Trade history | Background worker ingests `order_results/*.json` into Postgres via SSH/SFTP |

## Critical Compatibility Constraint Discovered

`simple_config_bot.py` (the existing entry point) **crashes** on missing `TELEGRAM_BOT_TOKEN` (lines 58–59 of [SellerMarket/simple_config_bot.py](SellerMarket/simple_config_bot.py)). We cannot just run it with no token. Solution: **override the container `command:`** in each agent stack's `docker-compose.yml` to bypass `simple_config_bot.py` entirely and run only the scheduler:

```yaml
command: ["python", "-u", "-c",
  "import logging,sys; logging.basicConfig(level=logging.INFO, stream=sys.stdout); from scheduler import JobScheduler; JobScheduler('/app/scheduler_config.json').run()"]
```

This requires **zero modification** to the existing app — only the compose file we generate. `python -u` ensures unbuffered stdout for live log streaming. The `logging.basicConfig` is necessary because `scheduler.py` itself never calls `basicConfig` ([SellerMarket/scheduler.py:16](SellerMarket/scheduler.py#L16)) — it relied on `simple_config_bot.py` to set it up.

## Data Model (PostgreSQL)

Tables (all timestamps `timestamptz`, all PKs `uuid`):

- `users(id, username UNIQUE, password_hash, role[admin|agent], telegram_user_id NULL, deleted_at, created_at)`
- `servers(id, name, host, ssh_port, ssh_user, ssh_auth[password|pubkey], ssh_secret_ref TEXT, host_key_pin TEXT, status, last_seen_at, base_dir DEFAULT '/root/seller-market/agents')`
  - `ssh_secret_ref` points to a file path on the mgmt VPS (`~/.ssh/sm_<server_id>`) for pubkey, OR a Fernet-encrypted blob for password. **Never store raw private keys in DB.**
  - `base_dir` defaults to `/root/seller-market/agents` — the **subdirectory** mgmt UI owns. The parent `/root/seller-market/` and its files remain manually managed by you (existing setup preserved).
- `settings(key PK, value, updated_by, updated_at)` — global key/value table. Admin sets `ocr_service_url = http://5.10.248.55:18080` here once; renderer reads it for every agent stack's `.env`.
- `agent_stacks(id, server_id, agent_id, stack_dir, compose_project, status[provisioning|up|down|deprovisioning], deployed_at, UNIQUE(server_id, agent_id))`
- `customers(id, agent_id, server_id NULL, stack_id NULL, assignment_status[pending|assigned|active], display_name, section_name UNIQUE GLOBAL, username, password_enc, broker, isin, side[1|2], enabled, comment, version INT, created_at, updated_at)`
  - `display_name` — agent-supplied friendly label (e.g. "Mostafa شفن خرید"); shown only in mgmt UI, never in `config.ini`.
  - `section_name` — auto-generated on INSERT, globally unique, immutable. Format: `a{agent_id}_c{customer_id}_{broker}_{isin}` (e.g. `a3_c42_bbi_IRO3AYHZ0001`). ISIN is included so the same person/account can hold multiple sections for different symbols. This is what appears as the `[section]` header in the rendered `config.ini`. Globally unique → no collisions across agents, servers, brokers, or ISINs.
  - `server_id` / `stack_id` — NULL until admin (or auto-policy) assigns the customer to a server. Until then, the customer is not in any `config.ini` on any server.
  - `password_enc` — Fernet-encrypted with split-key (see Security).
- `distribution_policies(id, scope[global|agent], agent_id NULL, policy[manual|round_robin|least_customers|broker_affinity], default_server_id NULL, updated_at)` — admin configures one global default + optional per-agent overrides. `manual` means admin must explicitly assign every customer.
- `scheduler_jobs(id, stack_id, name[cache_warmup|run_trading], time TIME, enabled, command, version)`
- `locust_configs(id, stack_id UNIQUE, users, spawn_rate, run_time, host, processes, version)`
- `runs(id, stack_id, agent_id, job_name, trigger[scheduled|manual|api|retry], started_at, finished_at, status[running|success|failed|killed], exit_code, log_blob_ref TEXT NULL, log_blob_sha256)`
- `trade_results(id, run_id, customer_id, tracking_number UNIQUE, isin, symbol, side, price, volume, executed_volume, state, state_desc, is_done, net_amount, created_at_broker, created_shamsi, raw_json JSONB, ingested_at)`
- `stack_run_locks(stack_id PK, run_id, kind[cache|trade], started_at, lease_expires_at, holder)` — exclusion for Q3 (manual vs scheduled collision)
- `ingest_cursors(stack_id PK, last_filename, last_mtime, updated_at)` — for the trade-result ingestor
- `health_signals(id, stack_id, kind, severity, message, raw, ts, ack_by, ack_at)` — broker 429s, captcha fails, 401s extracted from log
- `server_clock_skew_samples(server_id, sampled_at, delta_seconds)` — clock-drift monitoring
- `audit_log(id, actor_user_id, action, target_type, target_id, before_json JSONB, after_json JSONB, ts)`

## Concurrency Strategy

Three layers, each addressing a different race:

1. **Optimistic locking** via `version` column on `customers`, `scheduler_jobs`, `locust_configs`. UPDATEs use `WHERE id=? AND version=?` and bump `version`. Two operators editing the same row → second one gets `409 Conflict`.
2. **Postgres transactional advisory locks** for "render+push config files via SSH" — `pg_try_advisory_xact_lock(hash(stack_id))`. Serializes file pushes per stack. Different stacks push in parallel.
3. **Per-server compose lock** — `pg_try_advisory_xact_lock(hash('compose', server_id))` when running `docker compose up/down`. Docker daemon serializes its own state but compose project handling is fragile.

**Atomic remote writes:** SFTP write to `config.ini.tmp` → `mv -f config.ini.tmp config.ini && sync`. Single `rename(2)` syscall on same filesystem = atomic.

**Manual run vs scheduled run exclusion** (the big one):

- Before a manual run, INSERT into `stack_run_locks` (ON CONFLICT → 409 "another run is in progress").
- **Snapshot the scheduled job's `enabled` field, then set `enabled=false` in `scheduler_config.json` and SFTP it.** The scheduler picks this up within 1s via its polling re-read ([SellerMarket/scheduler.py:248-272](SellerMarket/scheduler.py#L248-L272)).
- Run the manual job via `docker exec -d` (detached) + a separate `docker logs -f` channel for streaming.
- On completion, restore `enabled` to its snapshot value, release the lock.

This avoids: log-file truncation race (locustfile_new.py truncates `trading_bot.log` on module import), broker double-orders, captcha session conflicts.

## Per-Agent Stack Deployment

**Non-destructive principle:** The mgmt UI **never** touches `/root/seller-market/` itself — your existing manual deployment (config.ini, Telegram bot, current Docker stack) stays exactly as it is. Mgmt UI only creates and operates inside `/root/seller-market/agents/<agent_id>/` subdirectories. You can continue SSH-editing the root-level config.ini and restarting the existing stack as before.

The mgmt UI provisions an agent stack on a server by:

1. SSH `mkdir -p /root/seller-market/agents/<agent_id>/{logs,order_results,.cache}`
2. SSH `touch trading_bot.log cache_warmup.log` (Docker bind-mount requires host files to exist)
3. SFTP-render the four files:
   - `docker-compose.yml` — agent stack defines **only** `trading-bot` (no OCR service). Uses the `command:` override above. Compose project name `sm-agent-<id>` ensures isolation from the existing root-level deployment.
   - `config.ini` (initially empty `[DEFAULT]` shell; populated as customers are added)
   - `scheduler_config.json` (with `enabled=false` initially — safe default)
   - `locust_config.json` (defaults: `users=10, spawn_rate=10, run_time=120s`)
   - `.env` — only `TZ=Asia/Tehran` and `OCR_SERVICE_URL=<value from settings table>` (e.g. `http://5.10.248.55:18080`). **No Telegram vars, no OCR sidecar.**
4. SSH `docker compose -p sm-agent-<id> -f /root/seller-market/agents/<agent_id>/docker-compose.yml up -d`
5. Update `agent_stacks.status = 'up'`

**OCR is global, not per-stack.** The OCR service already runs on its own server at `http://5.10.248.55:18080` (Swagger at `/swagger/index.html`). Admin declares this URL **once** in mgmt UI settings; every agent stack on every trading server points to it. The mgmt UI does not provision OCR. If the OCR server moves, admin updates one setting and the next config render propagates it. The existing root-level `/root/seller-market/` stack keeps using whatever OCR it already uses — we don't change it.

## Image & Port Strategy (Lightweight, Zero-Conflict)

**Image:**
- **Single shared image** for all agent stacks across all servers. Build it once, push to a registry (GHCR, Docker Hub, or a private registry), every agent stack references the same `image:` tag in compose. No per-agent build, no rebuild on customer change.
- **Base:** `python:3.11-slim` (matches the existing app's base). Multi-stage build: builder stage installs deps, final stage copies only the runtime artifacts. Strip docs/tests from site-packages. Expected size: ~200-300 MB.
- **Slim variant (optional, recommended):** ship a second tag `seller-market-scheduler:latest` whose `requirements.txt` excludes `pytelegrambotapi`, `flask`, `pyyaml`, `pytest` and anything related to OCR/EasyOCR (the agent stack never runs Telegram bot, never serves config_api.py, never decodes captchas locally). Realistically cuts image to ~150-200 MB. Same source tree, different requirements file at build time.
- Each agent's compose `image:` field is rendered by mgmt UI from a global setting (`agent_image_tag` in `settings` table) so you can roll out new image versions across all agents from one place.

**Ports — none, by design:**
- Agent stack `docker-compose.yml` defines **no `ports:` block** anywhere. Zero host port bindings, zero conflict possible — you can run 50 agent stacks on one server with no collisions.
- `locust` runs with `--headless` (no web UI port). It already does in your existing setup.
- OCR is remote, accessed by outbound HTTP — no local port.
- The scheduler is a Python process inside the container; all command triggering is via `docker exec` over SSH — no exposed port needed.
- All inter-container talk (if added later) goes over Docker's internal bridge network — agent stack uses `network_mode: bridge` (default) and creates no published ports.

**Future port allocation (v1.5+, only if needed):** if you ever add an in-container HTTP control plane to replace `docker exec`, add a `port_allocations(server_id, container_id, port UNIQUE per server)` table. Allocation atomically reserves the next free port in a configured range (e.g. 18100-18999) under a Postgres advisory lock per `server_id`. Avoids ever needing manual port assignment. Not required for v1.

## Customer Distribution Flow (Admin)

Agents never pick servers. The lifecycle is:

1. **Agent creates a customer** — enters `display_name`, `broker`, `username`, `password`, `isin`, `side`. Mgmt UI generates `section_name = a{agent_id}_c{customer_id}_{broker}` automatically. Row stored with `assignment_status='pending'`, `server_id=NULL`, `stack_id=NULL`. Customer is NOT yet in any `config.ini`.
2. **Admin's "Pending Customers" inbox** — shows every unassigned customer across all agents. Per customer, admin clicks "Assign to server X" (or the auto-policy resolver picks one based on `distribution_policies`).
3. **Auto-policy options** (admin-configurable, falls back to manual if unset):
   - `manual` — admin assigns every customer by hand. Safe default for sensitive setups.
   - `round_robin` — distributes new customers evenly across all online servers.
   - `least_customers` — assigns to the server with the fewest active customers.
   - `broker_affinity` — groups same-broker customers from the same agent onto the same server (shares captcha sessions and token cache, reduces broker rate-limit pressure).
4. **On assignment:**
   - Find-or-create the `(agent, server)` stack. If it's a brand-new stack, provision it (Phase 3 flow).
   - Set `customers.server_id`, `customers.stack_id`, `assignment_status='active'`.
   - Re-render that stack's `config.ini` from DB (all active customers for the stack) → advisory-locked SSH push.
   - Audit log entry.
5. **Reassignment / moves** — admin can move a customer from server A to server B. Mgmt UI: re-render both stacks' `config.ini`, push both, audit. If the customer was mid-trade, block the move until runs are idle (same gate as agent deletion).

**Agent view of distribution:** read-only badge on each of their customers ("Server: tehran-vps-2"). They do not get to change it. If they need a customer moved, they raise a request that surfaces in the admin inbox.

## Run-Now Flow (Agent or Admin)

1. Agent clicks "Run cache_warmup" on their stack:
2. Mgmt UI: acquire `stack_run_locks` row → 409 if busy.
3. Render the stack's current `config.ini` from DB (in case of pending edits) → advisory-lock-protected SSH push.
4. Snapshot scheduled jobs, set both jobs `enabled=false`, push `scheduler_config.json`.
5. SSH `docker exec -d sm-agent-<id>-bot python cache_warmup.py` (or `locust -f locustfile_new.py --headless --users N --spawn-rate R --run-time T --host H`).
6. Open WebSocket to UI; attach `docker logs -f --tail=200 sm-agent-<id>-bot` over SSH channel; stream to client.
7. Detect completion via container exec exit status OR via "Locust shut down" log marker; create `runs` row with status + exit_code; archive captured log to `log_blob_ref`.
8. Restore scheduled jobs to snapshotted `enabled` values, release `stack_run_locks`.
9. Kick ingestor for this stack — pulls any new `order_results/*.json` into `trade_results`.

## Scheduler Override Flow

Agent edits "run_trading time" from 08:44 to 08:50 in UI:

1. UPDATE `scheduler_jobs` with optimistic version check.
2. Render `scheduler_config.json` from DB.
3. Advisory-lock + SFTP atomic-write.
4. Scheduler polling loop picks up within 1s. **No reload trigger needed** — `JobScheduler.run()` re-reads on every iteration ([SellerMarket/scheduler.py:249](SellerMarket/scheduler.py#L249)).
5. UX detail: if the new time is later today AND the old one already fired, show a banner "This job will execute again at the new time today" — because `should_run_job` keys on `name+date+time_str` ([SellerMarket/scheduler.py:129](SellerMarket/scheduler.py#L129)) so a new time = new key = will re-fire.

## Trade-Result Ingestor

Per-stack background asyncio task, serialized via Postgres advisory lock keyed on `hash('ingest', stack_id)`:

1. `ls -1 --time-style=+%s order_results/*.json | sort` over SSH.
2. Prefilter by `mtime > last_mtime - 5s` (cheap), then precise filter `filename > last_filename` (per-account lexicographic — filename pattern is `{username}_{broker}_{YYYYMMDD_HHMMSS}.json`).
3. SFTP-fetch each new file, parse JSON, upsert `runs` row by (stack_id, file timestamp) → link to manual/scheduled run if present, else create synthetic run.
4. Bulk INSERT into `trade_results` with `ON CONFLICT (tracking_number) DO NOTHING`.
5. Update `ingest_cursors.last_filename` and `last_mtime` **in the same transaction** as the inserts.

Pool one Paramiko transport per `(server_id)`; serialize SFTP calls across stacks sharing a host.

**Janitor** (separate worker): after N days, SSH-delete ingested files from `order_results/`. The existing app never cleans them up.

## Security

- **Passwords (user logins):** bcrypt via `passlib`.
- **SSH credentials:** private keys stored as files at `~/.ssh/sm_<server_id>` (chmod 600), referenced by path in `servers.ssh_secret_ref`. Passwords (if used) Fernet-encrypted with **split-key**: half from env `MGMT_FERNET_KEY_PART1`, half from `/etc/sm/key.part2` (chmod 400). Both halves required to decrypt.
- **Broker passwords (customers.password_enc):** same split-key Fernet. Only the "render config.ini" code path decrypts — wrapped in a function that emits an `audit_log` entry on every decrypt.
- **`SecretStr` wrapper type** — never `str()`-able into logs or HTTP responses.
- **SSH host key pinning** via `servers.host_key_pin` — reject MITM after first connect.
- **Dedicated low-priv SSH user** on each trading server (not `root`); `sudo NOPASSWD` only for `docker compose -p sm-agent-*`, `docker exec sm-agent-*`, writes under `/root/seller-market/agents/`.
- **Role-based middleware:** every endpoint checks `current_user.role`. Every agent endpoint additionally scopes queries with `WHERE agent_id = current_user.id`.
- **CSRF tokens** on HTMX forms via `fastapi-csrf-protect` or equivalent.
- **JWT for WebSocket auth** — short-lived signed token in query string, verified on `ws.accept()`.
- **v2 milestone:** envelope encryption via Vault/KMS before 5th customer. Document this as a known limitation in v1.

## UI Surface

**Admin gets the union of admin-only screens PLUS every agent-scoped screen with an agent picker.**

**Admin-only:**
- **Settings** — global OCR URL (e.g. `http://5.10.248.55:18080`), Fernet keyset version, default locust caps
- **Servers** (CRUD) — name, host, SSH config, status, last seen, clock skew badge; "test connection" button (verifies SSH + verifies the existing `/root/seller-market/` is reachable but untouched)
- **Agents** (CRUD) — username/password, assign servers they can deploy to
- **Stacks** — list all agent stacks across all servers with status, deploy/redeploy/destroy actions
- **Global runs + audit log**
- **Health signals** (broker 429s, captcha fails) across all stacks

**Admin can act AS any agent** (same UI as the agent's own view, with an agent selector at the top):
- Manage any agent's customers
- Edit any agent's scheduler times
- Edit any agent's locust params (and is not bound by admin-set caps)
- Trigger run-now on any agent's stack
- View any agent's trade history and run logs

**Agent (scoped to `WHERE agent_id = current_user.id`):**
- **My customers** — declare `display_name`, broker, ISIN, side, credentials. Server is shown as a read-only badge (assigned by admin). New customers show `Pending assignment` until admin processes them. Agents cannot see server lists, SSH details, OCR URL, or other agents' data.
- **My stacks** — one row per server their customers landed on (may be 1, may be many). Each row shows: server name (read-only), status, scheduler editor (per-stack `scheduler_config.json`), locust params editor (bounded by admin caps).
- **Run cache_warmup / Run trading per stack** — confirmation dialog + dry-run preview. "Run on all my stacks" button fires them in parallel (each with its own lock).
- **My trade history** — across all stacks, filterable by date / broker / symbol / state.
- **My run logs** — live WebSocket + historical from blob.

**Universal:**
- Diff preview before any config push — render the proposed file, diff against current, require operator confirm. Cheap, prevents fat-fingered side flips.

## Risk Register (Prioritized)

| Pri | Risk | Mitigation |
|---|---|---|
| P0 | Manual + scheduled run collision (financial damage from double-orders, truncated logs) | Mutual exclusion via `stack_run_locks` + snapshot/disable `scheduler_config.json` |
| P0 | Compromise of mgmt VPS exposes every customer's broker password | Split-key Fernet for v1; envelope/KMS for v2 before 5th customer |
| P0 | SSH as `root` on trading servers | Dedicated low-priv user with scoped `sudo NOPASSWD` |
| P1 | Clock drift mgmt-VPS ↔ trading-server makes UI-shown times wrong; >120s drift silently skips scheduler jobs | NTP enforcement; periodic `date +%s` sample; UI warns at delta >5s |
| P1 | Empty `config.ini` (agent deletes last customer) → locust spawns 0 users and exits | UI must prevent removing the last enabled customer for a stack OR auto-disable the trade job |
| P1 | `locust.run_time > 600s` → killed mid-run by the 10-min subprocess timeout in scheduler ([SellerMarket/scheduler.py:227](SellerMarket/scheduler.py#L227)) | UI validation: `run_time < 10m`; warn otherwise |
| P2 | `processes=-1` on a 32-core box × N agents → CPU exhaustion | UI cap on `processes` per stack; stagger scheduled runs across stacks on same server |
| P2 | `trading_bot.log` truncated on every locust module import; can't tail post-hoc | Capture logs via WebSocket+archiver, not file tail; document |
| P2 | Filename collision in `order_results/{user}_{broker}_{ts}.json` if two runs fire same second | Mutual exclusion already prevents this; document as known limitation |
| P3 | Order results dir grows unboundedly (existing app never cleans up) | Janitor worker deletes ingested files after N-day retention |
| P3 | SSH dies mid-push (orphan `.tmp` files) | `mv -f` is atomic; janitor cleans stale `.tmp` files |
| P3 | Pre-existing app has no `config.ini` file locking | In agent stacks we are the only writer (no Telegram bot) → moot, but document |
| P1 | Mgmt UI accidentally writes to `/root/seller-market/` root (would clobber existing manual setup) | Path-validation guard in SSH service: reject any write whose path does not start with `<server.base_dir>/<agent_id>/`. Integration test asserts this. |
| P2 | Shared remote OCR (5.10.248.55:18080) goes down → every agent stack on every server fails authentication | Health worker pings OCR URL every minute; surfaces an admin banner; agents see "OCR unavailable" on run-now button. Admin can update OCR URL in settings without redeploying stacks (next config render picks it up; for immediate effect, `docker compose restart` the affected stacks). |

## Implementation Phases

Build in this order — each phase deliverable on its own.

1. **Bootstrap** — FastAPI skeleton, Postgres+Alembic, settings (incl. global OCR URL), bcrypt auth, base templates, login/logout, admin/agent role enforcement. Admin can impersonate-view any agent's scope.
2. **Servers + SSH** — Servers CRUD, Paramiko pool with host-key pinning, atomic SFTP write helper, "test connection" button (asserts existing `/root/seller-market/` is present and untouched), clock-skew sample.
3. **Stack provisioner** — Generate per-agent compose YAML with `command:` override and no OCR service (OCR_SERVICE_URL points to global setting), render+SFTP files into `agents/<id>/` only, `docker compose -p sm-agent-<id> up`, status polling. Provision/deprovision admin actions. **Guard: refuse to write any file at `/root/seller-market/` root level — only `agents/<id>/` paths allowed.**
4. **Customer CRUD + config.ini render+push** — Customers table with auto-generated `section_name` and `assignment_status` lifecycle. Agent UI takes only `display_name` + broker fields; mgmt UI generates the `[a{agent_id}_c{customer_id}_{broker}]` header. Optimistic locking, render service, advisory-lock-protected push, diff-preview UI, encryption of broker passwords. Admin "Pending Customers" inbox + manual assign. Auto-policy resolver (round_robin / least_customers / broker_affinity) as a follow-up within this phase. Customer move workflow (re-render both source and dest stacks).
5. **Scheduler + locust config editors** — Per-stack scheduler_jobs and locust_configs CRUD; render+push; validate `run_time<10m`, cap `processes`. UX banner for "will re-fire today".
6. **Run-now** — `stack_run_locks`, snapshot/disable scheduler dance, `docker exec -d`, WebSocket live log via `docker logs -f`, log archiver.
7. **Trade-result ingestor** — `ingest_cursors`, per-stack asyncio loop, advisory-locked SSH list+fetch+parse, `ON CONFLICT (tracking_number) DO NOTHING`. Trade history UI for agents (filter by date/broker/symbol/state). Admin global view.
8. **Health & janitor workers** — Periodic server heartbeat, clock-skew samples, tail `trading_bot.log` for 429/captcha regex → `health_signals`. Janitor deletes ingested files >N days.
9. **Audit log + dry-run preview** — Log every mutation with `before_json`/`after_json`. Show diff preview before any config push.
10. **Hardening** — CSRF tokens, JWT for WebSockets, rate limits on auth endpoints, secret rotation hook (`keyset_version` column), backup-encryption docs.

## Verification

End-to-end test plan:

1. **Unit:** `test_render_config.py` — DB rows ↔ INI/JSON files (golden-file diff against `SellerMarket/config.example.ini`); `test_ssh_atomic_write.py` — verify tmp+rename pattern via mocked SFTP.
2. **Integration:** spin up a real Docker container with the `command:` override; write a `scheduler_config.json` from a test harness; assert the scheduler picks up the change within 2s.
3. **Concurrency test:** two browsers (Playwright) editing the same customer simultaneously — assert second save gets 409 conflict.
4. **E2E happy path** (Docker-in-Docker testbed):
   - Admin creates server (use a throwaway VM or local Docker host as the "trading server")
   - Admin creates agent
   - Agent logs in, adds 2 customers (1 buy/1 sell)
   - Mgmt UI provisions agent stack on the server (verify `docker compose ps` shows `sm-agent-<id>-bot` up)
   - Agent clicks "Run cache_warmup" → live logs stream → exit 0
   - Agent edits trade time to "in 90 seconds" → enable → wait
   - Verify `order_results/*.json` lands on the server, ingestor picks it up, `trade_results` table populated
   - Verify agent's history page shows the trade
5. **Failure paths:**
   - Kill SSH mid-push → next attempt succeeds (cleans up `.tmp`)
   - Run manual cache_warmup while scheduler is enabled → second is blocked
   - Agent A cannot see Agent B's customers (verify with curl + JWT swap)
6. **Smoke deployment:** deploy mgmt_ui to a test VPS, point at one real trading server (the one at 5.10.248.55 in a sandbox slot), provision a test agent, run a non-trading-hour smoke run (use a fake `host` in `locust_config.json` so no real orders fire).
